5. race++ and race-- each compile to 3 instructions: load, modify, store.
two threads can both read the same value, modify it, then both write back,
so one update gets lost. max possible: +2,000,000 (all subtractions lost).
min possible: -2,000,000 (all additions lost).

7. yes, a race condition can still occur on a single-core machine. the OS
uses preemptive time-slicing and can context switch between the load and
store steps of race++/race--. stale register values get written back after
the switch. less likely than with true parallelism but still possible.

10. with 200M iterations the threads run for much longer, so the OS
preempts them many times. a thread can be interrupted after loading `race`
into a register, the other thread then runs and modifies `race` many times,
then the first thread resumes and writes its stale register value back,
wiping out all that work. enough context switches = race condition even on
one core.
