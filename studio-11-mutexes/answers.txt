8. when each thread locks once and holds the mutex for all 20M iterations,
the threads run fully sequentially with no contention. the loop runs entirely
in one thread's cache with no false sharing or lock overhead per iteration.
the racy version actually runs both threads simultaneously and causes cache
line bouncing between cores on every shared write, which is slower than
sequential. the per-thread lock avoids all of that.

10. per-iteration locking is appropriate when threads need fine-grained
interleaving - for example, multiple threads adding and removing items from
a shared queue where each insert/remove is independent. per-thread (bulk)
locking is appropriate when a thread needs to perform a large atomic batch
of work, such as updating many fields of a shared struct where intermediate
states would be inconsistent.
